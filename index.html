MCP как стандартная поверхность

MCP

MCP даёт общий протокол, через который AI-приложения могут работать с внешними системами предсказуемо и единообразно.

Tools

Действия и API-вызовы, которые модель может инициировать.

Resources

Данные и документы, которые можно читать как контекст.

Prompts

Готовые шаблоны и сценарии взаимодействия.

Почему это важно
  • Модель перестаёт жить только внутри prompt.
  • Внешние системы получают единый способ интеграции.
  • Появляется пространство для gateway, policy и orchestration.
Именно здесь начинается разговор не просто про tools, а про MCP gateway как новый слой системы.

MCP architecture

Роли
  • Host — держит UX, consent, orchestration и policy boundary.
  • Client — говорит по протоколу с сервером.
  • Server — публикует tools, resources и prompts.
Это уже не просто HTTP endpoint: тут есть роли, состояние и жизненный цикл взаимодействия.
Схема
flowchart LR H[Host] C1[Client A] C2[Client B] S1[MCP Server 1] S2[MCP Server 2] H --> C1 H --> C2 C1 --> S1 C2 --> S2

Proxy vs MCP gateway

Обычный proxy
flowchart LR U[User] --> A[LLM App] A --> P[Proxy] P --> S1[Tool Server A] P --> S2[Tool Server B]

Proxy умеет переслать запрос, но не понимает сессионную модель, capability negotiation и границы orchestration.

MCP gateway
flowchart LR U[User] --> H[Host] H --> C1[Client A] H --> C2[Client B] C1 --> S1[MCP Server A] C2 --> S2[MCP Server B]

MCP gateway понимает state, lifecycle, negotiated features и может собирать несколько downstream серверов в одну поверхность.

Главная мысль: MCP gateway — не reverse proxy, а stateful application layer.

Транспорт и сессия

Transport landscape
  • stdio — subprocess и прямой IO.
  • Streamable HTTP — JSON-RPC поверх HTTP.
  • SSE — server-to-client streaming и совместимость.
Что делает MCP сложнее
  • Session binding и lifecycle.
  • Initialize и capability negotiation.
  • Notifications, progress, logs.
  • Нужен boundary, который всё это понимает.
HTTP endpoint ≠ MCP gateway

Framework layer: FastMCP

Что даёт framework
  • Python primitives для MCP-приложений.
  • Серверы, proxy и gateway-паттерны.
  • Быстрый старт для experiments и внутренних платформ.
Это важный слайд для зала: MCP gateway не нужно собирать с нуля на голом протоколе.
Минимальный FastMCP

from fastmcp import FastMCP

mcp = FastMCP("Demo")

@mcp.tool
def echo(text: str) -> str:
    return text

Framework-слой превращает MCP из спецификации в инженерный конструктор.

Зачем это нужно
  • Хранить downstream session ids и negotiated state.
  • Держать route table для tool calls.
  • Буферизовать notifications и task state.
  • Быть единым местом правды для runtime.
Вот здесь MCP gateway окончательно перестаёт быть прокси и становится stateful application layer.

Initialize: как собирается единый мир

Handshake

async def initialize(session_id, client_caps):
    gw = GatewaySession(...)
    for name, server in registry.items():
        ds = await server.initialize(client_caps)
        gw.downstream[name] = ds
        for tool in ds.tools:
            public = f"{name}.{tool.name}"
            gw.tool_to_server[public] = name
            gw.visible_tools.append({...})
    sessions[session_id] = gw
    return {"tools": gw.visible_tools}
Что появляется после initialize
  • N downstream connections вместо одной локальной догадки.
  • Namespace вида `jira.create_issue`.
  • Capability snapshot на всю клиентскую сессию.
  • Единый tool catalog для клиента.

Route table вместо if-else ада

Идея
  • Имена инструментов уже подготовлены при initialize.
  • По tool name сразу находим нужный server.
  • Клиент не знает и не должен знать, где реально живёт tool.
Вызов

async def call_tool(session_id, tool_name, args):
    gw = self.get(session_id)
    server_name = gw.tool_to_server[tool_name]
    downstream = gw.downstream[server_name]
    raw_name = tool_name.split('.', 1)[1]
    return await downstream.call_tool(raw_name, args)

Transforms, grouping и policy

Transforms

Переименование, скрытие args, defaults, упрощение schema и description.

Это и есть product-facing сторона MCP gateway: не только вызвать tool, но и сформировать правильную поверхность для модели.

Event-Driven MCP Gateway

FastAPI, Redis Streams и Rust

Здесь локальный in-memory runtime эволюционирует в production-grade distributed system с message bus и отдельным IO worker layer.

API gateway → message bus → IO workers
FastAPI
stateless API, routing, transforms, auth
Redis Streams
request/response и notifications fan-in
Rust workers
stdio lifecycle, zero HTTP overhead, scale-to-zero

Почему локальный runtime ломается в production

stdio

MCP subprocesses требуют долгоживущего IO и плохо дружат с классическим request-per-worker подходом.

Sticky sessions

Stateful session model делает балансировку и failover сложнее, чем в обычном stateless API.

Долгие вызовы

Tool calls и notifications легко выбиваются из модели короткого HTTP request/response.
