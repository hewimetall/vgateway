От Base LLM к MCP

Почему одной “умной” модели недостаточно

История не про “LLM стали умнее”.
История про то, как они стали полезными.

Сначала индустрия учила модели следовать инструкциям и человеческим предпочтениям. Параллельно она учила их читать внешний контекст и вызывать инструменты. MCP появляется уже после этого — как стандартный способ подключать всё это без зоопарка интеграций.

2 параллельных трека

Схождение в 2023
Когда у агентов появились tools, стало ясно: проблема уже не только в модели, но и в способе интеграции.
Следующий шаг — протокол
MCP стандартизирует подключение инструментов и источников данных, а не изобретает новый тип агента.

Два способа расширить модель: RAG и Tools

RAG: докидываем знания
1. User query Пользователь задаёт вопрос, но в весах модели может не быть нужного знания или оно устарело.
↓
2. Retrieval Система ищет релевантные документы, чанки, записи из БД или внутренней knowledge base.
↓
3. Retrieved context Найденные фрагменты приклеиваются к промпту как дополнительный контекст.
↓
4. LLM answer Модель отвечает, опираясь на user intent + injected context.
Tools: докидываем действие и результат
1. User query Пользователь просит не только объяснить, но и что-то сделать или получить внешние данные.
↓
2. Tool schema В запрос модели заранее передаётся описание доступных инструментов: что они умеют и какие аргументы ждут.
↓
3. Tool call → execution Модель выбирает функцию, приложение исполняет её снаружи и получает результат.
↓
4. Tool result in context Результат вызова возвращается в следующий ход как новый контекст для модели.

Общая мысль

И RAG, и Tools — это по сути context augmentation. В одном случае в контекст добавляются найденные знания, в другом — описание инструментов и затем результат их выполнения.

user query + extra context → reasoning + answer

Ключевое различие

  • RAG приносит модели факты и документы.
  • Tools приносят модели возможность действовать и читать результат действия.
  • Оба подхода в итоге расширяют то, что модель “видит” в текущем контексте.
RAG добавляет retrieved information в prompt/context [web:23][web:26][web:29]. Function calling передаёт модели описание tools, затем результат выполнения возвращается в следующий запрос как часть многошагового tool-calling flow [web:18][web:15]. MCP формализует tools, resources и prompts как стандартные примитивы [web:3].

MCP — это стандартный транспорт

Что именно стандартизирует MCP

Anthropic представила Model Context Protocol в ноябре 2024 как открытый стандарт для подключения AI assistants к системам, где живут данные и инструменты.

Важная идея не в “новом типе интеллекта”, а в универсальном протоколе вместо разрозненных интеграций.

Tools

Что модель может сделать: вызвать API, триггернуть действие, запустить код.

Resources

Что модель может прочитать: файлы, БД, документы, внешний контекст.

Prompts

Как модель направлять: шаблоны, инструкции, заготовки поведения.

MCP — это стандартный транспорт

MCP для LLM-инструментов — примерно как HTTP для веб-сервисов.

Он не изобрёл tools, не изобрёл RAG и не изобрёл агентов. Он сделал главное: дал общий протокол, чтобы клиент и сервер могли договориться единообразно.

То есть шаг выглядит так: сначала появились useful models → потом у них появились tools → и только потом понадобился общий протокол для интеграции всего этого.

Переход к MCP — это не смена парадигмы reasoning, а стандартизация already-existing agent patterns.
Источники: Anthropic, “Introducing the Model Context Protocol”, 2024 [web:3]; материалы пользователя [file:1].

1. Архитектура MCP

Роли протокола

В MCP есть три ключевые роли: host, client и server; host управляет подключениями, client держит отдельную session к одному server, а server публикует primitives протокола [page:4][page:5].

  • host — orchestration, UX, consent, policy boundary [page:1].
  • client — protocol endpoint на одно server-соединение [page:1].
  • server — tools, resources, prompts [page:1].

2. Session lifecycle

Фазы session
1. Initialize
Client обязан начать с de>initialize и передать версию протокола, capabilities и сведения о себе [page:5].
2. Negotiation
Server отвечает своей версией, capabilities и de>serverInfo, после чего client отправляет de>notifications/initialized [page:5].
3. Operation
Рабочая фаза использует только согласованную версию протокола и только negotiated features [page:5].
4. Shutdown
Завершение делегировано транспорту: закрытию stdio или HTTP connection [page:5].

3. Capabilities и primitives

Capability negotiation

Capabilities определяют, какие части протокола реально доступны в текущей session, и выступают как feature gates [page:5].

  • Client: roots, sampling, elicitation, tasks [page:1].
  • Server: tools, resources, prompts, logging, completions, tasks [page:1].

5. Logging

Стандартизированная observability
  • Logging включается через server capability logging [page:1].
  • Client может выставлять минимальный log level через logging/setLevel [page:1].
  • Server отправляет log events через notifications/message [page:1].
  • Уровни следуют syslog severity model: от debug до emergency [page:1].

6. Authorization и transport

Transport-level security
  • Для HTTP-based transport авторизация в MCP строится вокруг OAuth 2.1-based подхода [page:2].
  • Client должен уметь находить auth server через Protected Resource Metadata, Authorization Server Metadata или OpenID Connect Discovery [page:2].
  • Access token передаётся через Authorization: Bearer header [page:2].
  • Для stdio рекомендуется использовать credentials из environment, а не переносить HTTP auth flow [page:2].
HTTP auth flow
flowchart LR
  C[Client] -->|resource metadata| R[Protected Resource]
  R -->|auth server metadata| A[Authorization Server]
  C -->|OAuth flow| A
  A -->|access token| C
  C -->|Authorization: Bearer| S[MCP Server over HTTP]
Идея: безопасность в MCP не пришивается к tool schema, а живёт на уровне transport и session access control [page:2].
Источники: MCP Authorization [page:2], reveal.js-mermaid-plugin usage [web:28]

Почему простой proxy не подходит

Что делает proxy
  • Пересылает запросы между моделью и внешним сервисом.
  • Обычно не управляет lifecycle session и feature negotiation [web:49][web:45].
  • Не даёт чёткой модели изоляции между разными server connections [web:45].
  • Плохо подходит для двусторонних protocol features вроде sampling и elicitation [web:3][web:5][web:52].
Что нужно MCP
  • Изолированные client-сессии в отношении 1:1 к server [web:45].
  • Capability negotiation на старте соединения [page:1].
  • Stateful session model, а не только request forwarding [web:49][page:1].
  • Контроль host над consent, security boundaries и context aggregation [web:45].
Наивный proxy
flowchart LR
  U[User] --> H[LLM App]
  H --> P[Proxy]
  P --> S1[Tool Server A]
  P --> S2[Tool Server B]

Proxy видит трафик, но сам по себе не задаёт стандартную session model, negotiated features и boundaries между серверами [web:45][web:49].

MCP host-client-server
flowchart LR
  U[User] --> H[Host]
  H --> C1[Client A]
  H --> C2[Client B]
  C1 <--> S1[MCP Server A]
  C2 <--> S2[MCP Server B]

Host управляет политиками и доступом, а каждый client держит отдельную stateful session с конкретным server [web:45][page:1].

Простой proxy решает только маршрутизацию, а MCP стандартизирует session lifecycle, capabilities, изоляцию, client-side features и контроль безопасности со стороны host [page:1][web:45][web:3].

8. Transport details

Транспорты MCP
  • Основные transport-варианты в актуальном MCP: stdio и Streamable HTTP [page:1].
  • SSE уже legacy по спецификации, но на практике его всё ещё нужно учитывать в compatibility и migration path [page:1].
  • stdio подходит для локального subprocess-взаимодействия через stdin/stdout [page:1].
  • Streamable HTTP использует POST/GET и может передавать server-to-client сообщения через SSE [page:1].
Transport landscape
flowchart TD
  A[MCP Client]
  B[stdio]
  C[Streamable HTTP]
  D[legacy SSE]

  A --> B
  A --> C
  A -. compatibility .-> D
  C --> E[SSE stream optional]

HTTP semantics

  • Каждое JSON-RPC сообщение client → server отправляется отдельным HTTP POST [page:1].
  • Client должен отправлять Accept: application/json, text/event-stream [page:1].
  • Server может ответить либо одним JSON response, либо SSE stream для нескольких сообщений [page:1].
  • Для server-to-client уведомлений client может открывать отдельный GET request [page:1].

Session и compatibility

  • Для stateful HTTP session server может вернуть MCP-Session-Id при инициализации [page:1].
  • Дальше client обязан передавать MCP-Session-Id и MCP-Protocol-Version в следующих запросах [page:1].
  • В переходный период клиенты и серверы часто держат fallback между Streamable HTTP и legacy SSE [page:1].
  • Новая разработка должна идти в сторону Streamable HTTP, а SSE стоит рассматривать как compatibility layer [page:1].
Идея слайда: transport в MCP — это не просто канал доставки, а правила для JSON-RPC framing, streaming, session management, compatibility и безопасной эксплуатации [page:1].
Источники: MCP Custom Transports / legacy compatibility [page:1], MCP TypeScript SDK transport compatibility examples [page:2]

Web / Streamable HTTP

Здесь gateway уже не proxy, а session-aware MCP endpoint

Что происходит
Один HTTP MCP endpoint для клиента
Клиент работает с gateway как с одним MCP server поверх Streamable HTTP.
Одна внешняя session boundary
После initialize gateway ведёт внешнюю клиентскую сессию и связывает её с несколькими downstream MCP sessions.
Fan-in событий и streaming
Notifications, progress, logging и другие server messages нужно собрать обратно в один клиентский поток.
Capability negotiation проходит через gateway
Наружу публикуются только те возможности, которые gateway реально умеет поддержать end-to-end.
Почему не proxy
Lifecycle
Gateway должен понимать initialize, initialized и последующую сессионную фазу, а не просто пересылать HTTP.
MCP-Session-Id
Если session id выдан, его нужно удерживать и использовать как ключ к внутреннему session mapping.
State store
Нужно хранить client session id, downstream session ids, negotiated capabilities и route table.
Итог
Обычный reverse proxy не знает ничего про MCP session semantics, поэтому этого уровня недостаточно.
Главный тезис: в web-режиме vmcp-gateway обязан быть session-aware, потому что MCP поверх HTTP остаётся stateful.
В Streamable HTTP сервер может выдавать MCP-Session-Id, а клиент должен использовать его в последующих запросах. Lifecycle и capability negotiation являются обязательной частью протокола, поэтому gateway должен хранить session mapping и управлять событиями, а не быть простым proxy. [web:20][web:19][file:1]

Группировка MCP: варианты решения

local, web, kubernetes

1. Local / stdio

Gateway живёт рядом с клиентом, агрегирует несколько MCP server в один локальный endpoint, а состояние сессии держит в памяти процесса.

2. Web

Gateway становится сетевым MCP endpoint: держит session mapping, поток событий, capability negotiation и transport-level auth для HTTP.

3. Kubernetes

Gateway работает как production-layer: multi-tenancy, routing, auth boundary, session store и изоляция инструментов по namespace.

много MCP server → один управляемый endpoint
MCP задаёт lifecycle, capability negotiation, standard transports и transport-level authorization для HTTP-based сценариев; grouping нужен как session-aware слой поверх нескольких серверов. [web:19][web:20][web:25][web:8]

Самый простой способ сгруппировать несколько MCP server

Что происходит

В HTTP auth становится частью архитектуры gateway

Роль gateway

  • Один вход для security и audit.

От local-агрегации к production-платформе

FastMCP: быстрый runtime для MCP

Что это даёт
  • Python framework для MCP applications.
  • Tools описываются обычными Python-функциями.
  • Подходит, чтобы быстро собрать server, proxy или gateway.
Для доклада это важный мост: от спецификации MCP к реально работающему прототипу.
Минимальный пример

Из обычной функции получается MCP tool со схемой и валидацией.

FastMCP позиционируется как Python framework для MCP applications; tools можно объявлять обычными Python-функциями, а framework автоматически строит schema, validation и documentation. Tool Transforms в FastMCP

Что можно менять
  • Переименовывать tools и упрощать descriptions.
  • Переименовывать, скрывать и задавать default для аргументов.
  • Подменять transform function и контролировать structured output.
Это нужно, когда исходный tool неудобен для LLM, но менять сам backend или upstream server не хочется.
Базовые built-in transforms
  • Namespace — префиксы имён при композиции серверов.
  • Tool Transformation — rename, description, reshape args.
  • Enabled — скрываем лишние компоненты во время runtime.
  • Tool Search — вместо большого каталога оставляем поиск.
  • Resources as Tools / Prompts as Tools — адаптация под tool-only clients.
FastMCP Transforms Overview перечисляет built-in transforms: Namespace, Tool Transformation, Enabled, Tool Search, Resources as Tools, Prompts as Tools и Code Mode. Tool transformation поддерживает rename, argument reshaping и изменение описаний. [gofastmcp](https://gofastmcp.com/python-sdk/fastmcp-tools-tool_transform)

Пример трансформации tool

До
@mcp.tool
def verbose_internal_data_fetcher(
    query: str,
    include_metadata: bool = True,
    internal_limit: int = 100
) -> str:
    return "..."
Для LLM это слишком длинно, шумно и внутренне-ориентированно.
После
from fastmcp.server.transforms import ToolTransform
from fastmcp.tools.tool_transform import ToolTransformConfig

mcp.add_transform(ToolTransform({
  "verbose_internal_data_fetcher": ToolTransformConfig(
    name="search",
    description="Search the database."
  )
}))
verbose_internal_data_fetcher → search
FastMCP Tool Transformation позволяет переименовывать tools и менять descriptions, в том числе для tools из mounted servers или proxies. `ToolTransformConfig` применяется к исходным именам tool’ов и меняет то, что видит клиент. [gofastmcp](https://gofastmcp.com/servers/transforms/tool-transformation)

Аргументы тоже можно сжать

До
@tool
def search(q: str, limit: int = 10) -> list[str]:
    return [f"Result {i} for {q}" for i in range(limit)]
  • q — слишком короткое и неочевидное имя.
  • limit можно скрыть или зафиксировать default.
После
from fastmcp.tools import Tool
from fastmcp.tools.tool_transform import ArgTransform

better_search = Tool.from_tool(
    search,
    name="find_items",
    description="Find items matching your query.",
    transform_args={
        "q": ArgTransform(name="query"),
        "limit": ArgTransform(default=5, hide=True),
    },
)
  • Понятнее для модели.
  • Меньше schema noise.
  • Меньше соблазна у модели трогать лишние параметры.
`Tool.from_tool()` и `ArgTransform` в FastMCP позволяют переименовывать аргументы, задавать defaults, скрывать параметры и менять описание схемы без изменения исходного tool. [gofastmcp](https://gofastmcp.com/v2/patterns/tool-transformation)

Компрессоры: как сжать каталог tools

Идея
  • Проблема не только в данных, но и в самих описаниях tools.
  • Большой MCP server может тратить десятки тысяч токенов только на catalog.
  • Компрессор заменяет большой набор tool descriptions на маленький wrapper layer.
Для gateway-сценариев это прямое снижение контекстной стоимости.
Что говорит практика
  • GitHub MCP server: 94 tools и около 17,600 tokens только на описания.
  • Atlassian MCP server: около 10,000 tokens.
  • Wrapper-подход может сократить это на 70–97%.
17,600 → 528–5,280 tokens
MCP Compressor описывает проблему больших tool catalogs: GitHub MCP server с 94 tools потребляет около 17,600 tokens, Atlassian MCP server — около 10,000 tokens только на tool descriptions. Тот же источник заявляет снижение потребления на 70–97% за счёт замены десятков tools на 2 wrapper tools. [atlassian-labs.github](https://atlassian-labs.github.io/mcp-compressor/)

Сколько можно сэкономить

Быстрый счёт
  • Если каталог занимает 17,600 tokens, то при сжатии на 70% остаётся 5,280 tokens.
  • При сжатии на 90% остаётся 1,760 tokens.
  • При сжатии на 97% остаётся всего 528 tokens.
Для доклада главный вывод
  • Экономия идёт не на единицы процентов, а на порядок.
  • Особенно полезно, когда у gateway десятки или сотни tools.
  • Tool compression — это уже optimisation layer поверх MCP composition.
То есть после “как подключить tools” следующий зрелый вопрос — “как не утонуть в их контекстной цене”.
На примере 17,600 tokens сокращение на 70–97% даёт диапазон примерно от 5,280 до 528 tokens. Это соответствует общей задаче уменьшения стоимости tool catalogs в MCP-сценариях. [atlassian-labs.github](https://atlassian-labs.github.io/mcp-compressor/)

Сжатие запроса перед LLM

Идея
  • Запрос можно мутировать до отправки в модель.
  • Сжимаем длинные descriptions, chat context и tool hints.
  • Выбираем стратегию под клиента: quality, balanced, aggressive.
Это уже не просто tool transformation, а отдельный compression layer перед inference.
Что можно использовать
  • LLMLingua — model-based prompt compression.
  • Selective Context — оставляет только информативные части.
  • emoji — кастомный shorthand для контролируемых сценариев.
`llmlingua` доступен как Python-пакет для prompt compression. `Selective Context` сжимает prompt/context, оставляя наиболее информативные части. Пакет `emoji` позволяет делать shortcode/emoji преобразования для кастомного сокращения текста. [github](https://github.com/liyucheng09/Selective_Context)

Пример мутации запроса

До
Исходный запрос Summarize the last 20 file operations, explain repeated failures, include metadata, preserve all warnings, and suggest next debugging steps.
Хороший для человека, но дорогой как runtime context.
После
Сжатый вариант Summarize recent file ops, repeated failures, warnings, next debug steps.
semantic preserve → fewer tokens
LLMLingua поддерживает `compress_prompt()` с управлением через `target_token`. Selective Context также предназначен для сжатия prompt/context без потери ключевой информации. [github](https://github.com/microsoft/LLMLingua/blob/main/llmlingua/prompt_compressor.py)

Пример на Python

LLMLingua
from llmlingua import PromptCompressor

compressor = PromptCompressor()
result = compressor.compress_prompt(
    context=[user_prompt],
    target_token=80
)

compressed = result["compressed_prompt"]
Emoji shorthand
text = user_prompt
text = text.replace("warning", "⚠️")
text = text.replace("critical", "🔥")
text = text.replace("success", "✅")

Не универсальный compressor, а кастомный shorthand для контролируемых клиентов.

`PromptCompressor.compress_prompt(..., target_token=...)` показан в коде LLMLingua. Пакет `emoji` и связанные примеры подтверждают преобразование текста и shortcode/emoji representation. [hrekov](https://hrekov.com/blog/remove-replace-emoji-python)

Динамический выбор компрессии

По типу клиента
  • Rich client → мягкая компрессия через LLMLingua.
  • Lightweight client → aggressive pruning.
  • Domain client → кастомные alias/emoji/rules.
Смысл
  • Сжатие становится policy layer.
  • Один и тот же tool catalog можно отдавать по-разному.
  • Экономим токены без жёстко одного универсального режима.
Не просто “сжать всё”, а “сжать по профилю клиента”.
FastMCP поддерживает transforms как отдельный слой адаптации серверного интерфейса. Python-библиотеки вроде `llmlingua`, `selective-context` и rule-based approaches позволяют строить client-aware compression policies поверх этого слоя. [pypi](https://pypi.org/project/llmlingua/)

Краткая реализация сжатия на FastMCP

Идея
  • Принимаем длинный запрос от клиента.
  • Перед вызовом upstream tool прогоняем его через компрессор.
  • Логируем, сколько символов или токенов удалось убрать.
В FastMCP это можно делать либо как отдельный tool-wrapper, либо через middleware around request flow.
Где встраивать
  • Tool transform — меняем интерфейс и аргументы tool.
  • Middleware — перехватываем request и мутируем payload централизованно.
  • Context — пишем logs и сохраняем статистику сжатия.
FastMCP поддерживает Tool Transformation для изменения tool schemas и поведения. Middleware в FastMCP может inspect/modify requests in pipeline. Context даёт logging и state внутри tool execution. [gofastmcp](https://gofastmcp.com/v2/patterns/tool-transformation)

Пример: wrapper tool с компрессией

Код
from fastmcp import FastMCP, Context
from llmlingua import PromptCompressor

mcp = FastMCP("Compression Demo")
compressor = PromptCompressor()

@mcp.tool
async def compressed_search(query: str, ctx: Context) -> str:
    result = compressor.compress_prompt(
        context=[query],
        target_token=80
    )
    compressed = result["compressed_prompt"]

    await ctx.info(f"query: {len(query)} -> {len(compressed)} chars")
    await ctx.set_state("last_compressed_query", compressed)

    return compressed
Что здесь происходит
ВходДлинный пользовательский query
↓
LLMLinguaСжимает текст до target budget
↓
ContextЛогируем и сохраняем результат
↓
ВыходКороткий query для следующего tool/LLM call
LLMLingua предоставляет `PromptCompressor` и `compress_prompt()` для prompt compression. FastMCP Context позволяет логировать через `ctx.info(...)` и сохранять session state через `ctx.set_state(...)`. [gofastmcp](https://gofastmcp.com/python-sdk/fastmcp-server-context)

Вариант с client-aware стратегией

Простая стратегия
def compress_for_client(text: str, client: str) -> str:
    if client == "rich":
        return llmlingua_compress(text)
    if client == "lite":
        return shrink_prompt(text)
    if client == "emoji":
        return to_emoji_shortcuts(text)
    return text

Один и тот же запрос сжимается по-разному в зависимости от клиента.

Зачем это нужно
  • Desktop client может позволить более мягкое сжатие.
  • Лёгкий web client — aggressive compression.
  • Domain-specific client — custom shorthand или aliases.
Compression становится policy layer, а не одной жёсткой функцией для всех.
Python-пакеты `llmlingua`, `shrink-prompt` и `emoji` подходят для разных стратегий сжатия. FastMCP transforms дают естественный слой, поверх которого можно строить client-aware adaptation. [gofastmcp](https://gofastmcp.com/servers/transforms/transforms)

    GW --> M1
    GW --> M2
    GW --> M3
    GW --> M4
    REG -. catalog / discovery .-> GW
Этот слайд заменяет длинный хвост про production-детали и сразу показывает итоговую систему целиком.

Поверх gateway — FastAPI BFF

Зачем ещё один слой
Из tools в product API
Клиенту не нужен tool catalog
Продукт и UI работают с бизнес-операциями, а не с сырыми MCP tools.
BFF даёт нормальный контракт
REST/HTTP endpoint, валидация, auth, rate limits, audit и versioning.
Gateway остаётся внутри
FastAPI оркестрирует сценарий, а cloud gateway уже вызывает нужные MCP capabilities. 
from fastapi import FastAPI, Depends
from pydantic import BaseModel

app = FastAPI()

class CreateIssueRequest(BaseModel):
    project: str
    title: str
    description: str | None = None

class GatewayClient:
    async def call_tool(self, tool: str, **kwargs):
        ...

async def get_gateway() -> GatewayClient:
    return GatewayClient()

@app.post("/api/issues")
async def create_issue(
    body: CreateIssueRequest,
    gw: GatewayClient = Depends(get_gateway),
):
    return await gw.call_tool(
        "jira.create_issue",
        project=body.project,
        summary=body.title,
        description=body.description,
    )
Снаружи у нас продуктовый API. Внутри — FastAPI, cloud gateway, registry и MCP-серверы.

Ключевой финал

Что было бы плохо

  • Тащить клиента напрямую в десятки tools.
  • Раздавать много MCP endpoint’ов.
  • Смешивать product API и integration layer.
  • Держать архитектуру на ручной сборке коннекторов.
→

Что выглядит зрелым

  • FastAPI как внешний BFF/API слой.
  • FastMCP Cloud Gateway как единая MCP точка входа.
  • Registry как каталог и discovery-слой.
  • MCP servers как execution layer реальных интеграций.
То есть конец доклада не про “ещё один framework”, а про зрелую форму системы: product API сверху, gateway посередине, MCP-интеграции снизу.
